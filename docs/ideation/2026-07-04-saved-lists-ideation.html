Saved Lists — Ideation

A personal, cloud-backed way to save catalog problems into named lists — collaboration comes later · 4 Jul 2026

Subject: a new feat/saved-lists feature, off main Trigger: the collaborative-first Lists work (PR #10) felt heavy and wrong to use Method: grilling session (grill-me) — resolving the design tree one branch at a time Grounding: in-session

The one thing to take away

PR #10 built lists collaborative-first — every list a cloud, membership-scoped, shareable object, with "personal" as a degenerate one-member case. That bundled the simplest, most-used act — save a problem to a list — with all the multiplayer machinery. The reframe: keep #10's backend (it's sound) but extract it onto a fresh feature off main and ship the personal cloud list first, deferring everything collaborative to a later layer.

This isn't a "redesign" of #10 — it's a new feature on a clean base. The app has no Lists tab on main at all; Saved Lists adds one. #10 is superseded as the delivery vehicle: its backend is extracted here, its collaborative UI revived later.

Now — Saved Lists

Cloud lists (a personal list = a one-member list): create / rename / delete, add & remove problems, plus a local Favorites view. Signed in, synced across your devices via the account you already use for the logbook.

Later — Collaboration

Apply the deferred migrations and un-hide what #10 already built: sharing, member roster, one-way promotion, group status. Mostly a revival, not a rewrite. Its own future plan.

Why extract #10's backend, ship personal-first

The pain — a collaborative-first product bundles the 95% (solo curation) with the 5% (a co-located group). Re-sequencing ships the useful core first.
The inversion — treat the personal list as the primary object and collaboration as an enhancement of it. A list becomes shareable when you choose; until then it's just yours.
Reuse over rewrite — #10's schema (lists/list_members/list_problems + RLS) and ListsManager are solid. Extract the backend; leave the join/group-status RPCs (0004/0005) and sharing UI dormant until the collaboration layer.
Accepted tradeoff — cloud lists need sign-in (like the cloud logbook); the tab shows a prompt when signed out. Favorites stays local and needs no auth. The auth + Supabase foundation is already on main (merged PR #8).

Carried forward for the later collaboration layer

Decided in the earlier grill (when the feature was collaborative-first); still true, deferred to the cloud phase:

Personal vs shared is explicit — a list becomes collaborative by an explicit action, not by who happens to be in it.
Sharing is opt-in and one-way — a private list stays private until you Share it, which promotes it (no silent demotion).
Member presence on cards — shared lists show member avatars + count.
Open (later): demotion of a shared list back to private — currently one-way. Revisit in the collaboration plan.
